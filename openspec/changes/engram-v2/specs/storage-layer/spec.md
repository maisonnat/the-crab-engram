# Storage Layer Specification

## Purpose

Definir el Storage trait abstracto (firewall contra vendor lock-in) y la implementación SQLite con FTS5, vector search, y 16 migrations versionadas.

## Requirements

### Requirement: Storage Trait Abstraction

The system MUST define a Storage trait that ALL code depends on. No rusqlite types may leak outside `crates/store`.

#### Scenario: All methods return core types

- GIVEN any Storage trait method
- WHEN inspecting return types
- THEN all types are defined in `crates/core` (not rusqlite types)

#### Scenario: Error type is EngramError

- GIVEN any Storage trait method
- WHEN it fails
- THEN it returns `Result<T, EngramError>` (not `rusqlite::Error`)

#### Scenario: No raw SQL access

- GIVEN the Storage trait
- WHEN inspecting method signatures
- THEN no method accepts SQL strings or returns raw connections

### Requirement: SQLite Configuration

The system MUST configure SQLite for concurrent-safe, performant operation.

#### Scenario: WAL mode enabled

- GIVEN SqliteStore is initialized
- WHEN connection is established
- THEN journal_mode=WAL is set

#### Scenario: Busy timeout prevents lock contention

- GIVEN concurrent read/write operations
- WHEN lock contention occurs
- THEN busy_timeout=5000 waits before failing

### Requirement: Schema Migrations

The system MUST version all migrations and apply them idempotently.

#### Scenario: First run applies all migrations

- GIVEN empty database
- WHEN SqliteStore::new() is called
- THEN all migrations 001-016 are applied in order

#### Scenario: Second run is no-op

- GIVEN database with all migrations applied
- WHEN SqliteStore::new() is called
- THEN no migrations are applied, no errors

#### Scenario: Migrations tracked in _migrations table

- GIVEN migration 005 applied
- WHEN _migrations table is queried
- THEN row with version=5 exists

### Requirement: Observation CRUD

The system MUST support create, read, update, delete operations on observations.

#### Scenario: Insert with dedup hash

- GIVEN observation content
- WHEN inserted
- THEN SHA-256 normalized_hash is computed, duplicate within 15min window is rejected

#### Scenario: Upsert by topic_key

- GIVEN existing observation with topic_key="auth/config"
- WHEN new observation with same topic_key is inserted
- THEN existing is updated (not duplicated)

#### Scenario: Get increments access tracking

- GIVEN observation with access_count=5
- WHEN observation is retrieved by ID
- THEN access_count=6 and last_accessed=now

#### Scenario: Soft delete preserves data

- GIVEN observation is deleted (soft)
- WHEN queried directly by ID
- THEN observation is returned with deleted_at set

### Requirement: FTS5 Index

The system MUST maintain a full-text search index synchronized with observations.

#### Scenario: FTS index updates on insert

- GIVEN observation inserted with content "JWT auth with RS256"
- WHEN FTS5 MATCH "JWT" is performed
- THEN observation appears in results

#### Scenario: FTS index updates on delete

- GIVEN observation in FTS index
- WHEN observation is deleted
- THEN FTS entry is removed via trigger

#### Scenario: FTS5 special characters are escaped

- GIVEN query with characters like "C++" or "N+1"
- WHEN query is normalized
- THEN special characters are escaped for FTS5 MATCH

### Requirement: Vector Storage

The system MUST store 384-dimensional embeddings with model metadata.

#### Scenario: Embedding stored with model version

- GIVEN embedding generated by all-MiniLM-L6-v2
- WHEN stored
- THEN embedding_meta table records model_name="all-MiniLM-L6-v2", model_version="v2"

#### Scenario: Vector search returns similar observations

- GIVEN stored embedding for "performance optimization"
- WHEN vector search for "speed improvement" is performed
- THEN original observation appears in results with high similarity

### Requirement: Graph Operations

The system MUST support temporal graph operations on edges.

#### Scenario: Add edge with temporal auto-closure

- GIVEN edge A→B (RelatedTo) exists
- WHEN new edge A→B (RelatedTo) is created
- THEN old edge valid_until=now, new edge valid_from=now

#### Scenario: Get edges returns only active

- GIVEN 3 edges from node A (2 active, 1 superseded)
- WHEN get_edges(A) is called
- THEN only 2 active edges returned

#### Scenario: Get edges at specific time

- GIVEN edge valid_from=T1, valid_until=T3
- WHEN get_edges_at(node, T2) is called (T1 < T2 < T3)
- THEN edge is returned

### Requirement: Export/Import Compatibility

The system MUST export/import JSON compatible with Engram Go.

#### Scenario: Round-trip preserves data

- GIVEN observations exported to JSON
- WHEN imported into fresh database
- THEN all observations, sessions, and prompts are preserved

#### Scenario: Go format compatibility

- GIVEN export from Engram Go
- WHEN imported into engram-rust
- THEN data is accepted without errors

### Requirement: Lifecycle State Transitions

The system MUST support batch state transitions by type and age.

#### Scenario: Batch transition Active→Stale

- GIVEN 50 Command observations older than 30 days in Active state
- WHEN transition_state(project, Command, "active", "stale", 30) is called
- THEN 50 observations transition to stale, count returned

#### Scenario: Batch transition respects policy

- GIVEN Decision observations of any age
- WHEN lifecycle transitions run
- THEN zero Decisions transition (permanent type)

### Requirement: Belief Storage

The system MUST store beliefs with state machine and history.

#### Scenario: Belief CRUD with state transitions

- GIVEN belief created with state=Active
- WHEN confirmed by additional evidence
- THEN confidence increases, state may become Confirmed

#### Scenario: Previous values preserved on update

- GIVEN belief with current_value="RS256"
- WHEN updated to "ES256"
- THEN previous_values contains {"RS256", valid_from, valid_until, superseded_by}

### Requirement: Entity Storage

The system MUST store entities with aliases and mention tracking.

#### Scenario: Entity with multiple aliases

- GIVEN entity "Alice" with aliases ["our CTO", "the reviewer"]
- WHEN searched by any alias
- THEN entity is found

#### Scenario: Entity mentions link to observations

- GIVEN entity mentioned in 3 observations
- WHEN get_observations_by_entity is called
- THEN all 3 observations are returned
