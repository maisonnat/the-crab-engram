# Phase 1, Plan 01: Expand Build Matrix to 8 Targets

## Objective

Expand `.github/workflows/release.yml` from 3 targets / 3 artifacts to 8 targets / 12 artifacts with target-triple naming, cross-compilation for all Linux targets via `cross`, and a separate .deb packaging job.

**Purpose:** The self-update engine (Phase 2) needs archives named with full target triples for asset discovery via `self_update` crate. This phase establishes the build infrastructure that all downstream phases depend on.

**Output:** Updated `.github/workflows/release.yml` with 8-target build matrix, cross-compilation setup, .deb job, and updated release job.

## Context

@.github/workflows/release.yml
@Cargo.toml
@.planning/phases/01-build-matrix/01-CONTEXT.md

### Current State

The release workflow builds 3 targets:
- `x86_64-unknown-linux-gnu` on `ubuntu-latest` with native `cargo`
- `aarch64-apple-darwin` on `macos-latest` with native `cargo`
- `x86_64-pc-windows-msvc` on `windows-latest` with native `cargo`

Archive naming uses friendly names (e.g., `the-crab-engram-linux-x86_64.tar.gz`) — **must change** to target-triple format for `self_update` compatibility.

**Archive naming (target-triple format, per D-04):**
- Unix: `the-crab-engram-${{ github.ref_name }}-${{ matrix.target }}.tar.gz`
- Windows: `the-crab-engram-${{ github.ref_name }}-${{ matrix.target }}.zip`

**Steps to modify:**

1. **Add cross installation step** (before Rust install, only for Linux targets):
```yaml
      - name: Install cross (Linux)
        if: matrix.use_cross
        run: |
          CROSS_URL="https://github.com/cross-rs/cross/releases/download/v0.2.5/cross-x86_64-unknown-linux-musl.tar.gz"
          curl -sSL "$CROSS_URL" | tar xz -C /usr/local/bin
```

2. **Build step** — change from `cargo build` to conditional:
```yaml
      - name: Build release
        run: |
          if [ "${{ matrix.use_cross }}" = "true" ]; then
            cross build --release --target ${{ matrix.target }}
          else
            cargo build --release --target ${{ matrix.target }}
          fi
```
For Windows builds (where this step runs on Windows runners), use a different shell. Add `shell: bash` to make it work uniformly, OR split into two conditional steps:
```yaml
      - name: Build release (cross)
        if: matrix.use_cross
        run: cross build --release --target ${{ matrix.target }}

      - name: Build release (cargo)
        if: "!matrix.use_cross"
        run: cargo build --release --target ${{ matrix.target }}
```

3. **Packaging steps** — update archive names to use target triple:
```yaml
      - name: Package (Unix)
        if: runner.os != 'Windows'
        run: |
          cd target/${{ matrix.target }}/release
          tar czf ../../../the-crab-engram-${{ github.ref_name }}-${{ matrix.target }}.tar.gz the-crab-engram

      - name: Package (Windows)
        if: runner.os == 'Windows'
        shell: pwsh
        run: |
          Compress-Archive -Path "target/${{ matrix.target }}/release/the-crab-engram.exe" -DestinationPath "the-crab-engram-${{ github.ref_name }}-${{ matrix.target }}.zip"
```

4. **Upload artifact** — update name and path to match new naming:
```yaml
      - name: Upload artifact
        uses: actions/upload-artifact@v4
        with:
          name: archive-${{ matrix.target }}
          path: |
            *.tar.gz
            *.zip
```

**IMPORTANT:** Keep existing `env: BINARY_NAME: the-crab-engram`. Keep `actions/checkout@v4`, `dtolnay/rust-toolchain@stable`, `actions/cache@v4`. The cache key already uses `${{ matrix.target }}` — no change needed there.

For the `actions/cache@v4` step: when using `cross`, the cache paths should also include the cross data directory. But for simplicity, keep the existing cache config — cross uses its own Docker-based build and the cache will still help with registry data.

Add a new job `build-release-deb` to `.github/workflows/release.yml`, placed between the `build` job and the `release` job. This job runs independently (not as part of the build matrix). Per D-05, pattern from ripgrep's `build-release-deb`.

```yaml
  build-release-deb:
    name: Build .deb package
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Rust
        uses: dtolnay/rust-toolchain@stable
        with:
          targets: x86_64-unknown-linux-gnu

      - name: Install cargo-deb
        run: cargo install cargo-deb

      - name: Build .deb
        run: cargo deb --target x86_64-unknown-linux-gnu

      - name: Rename .deb with version
        run: |
          cd target/x86_64-unknown-linux-gnu/debian
          DEB_FILE=$(ls *.deb | head -1)
          mv "$DEB_FILE" "the-crab-engram-${{ github.ref_name }}-x86_64-unknown-linux-gnu.deb"

      - name: Upload .deb artifact
        uses: actions/upload-artifact@v4
        with:
          name: deb-package
          path: target/x86_64-unknown-linux-gnu/debian/*.deb
```

**Why separate job:** Follows ripgrep pattern. Keeps the build matrix focused on compilation. The .deb job can run in parallel with the matrix builds. Also avoids polluting the matrix with packaging-specific steps.

**Note on `cargo-deb` metadata:** The `Cargo.toml` root `[package]` section has `name = "engram-rust"` and `[[bin]] name = "the-crab-engram"`. `cargo-deb` will use the binary name from `[[bin]]`. If `cargo-deb` fails due to missing `[package.metadata.deb]`, it uses sensible defaults — this is acceptable for Phase 1.
