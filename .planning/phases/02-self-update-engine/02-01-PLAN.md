    The function must implement this exact flow:

    1. **Check for update** — use `ReleaseList::configure()` for `--check-only` and `--dry-run`
       - Fetch latest release version
       - Compare with `self_update::cargo_crate_version!()`
       - If `--check-only`: print "Latest: vX.Y.Z (current: vX.Y.Z)" and return
       - If `--dry-run`: print "Would update from vX.Y.Z to V.W.Z" and return
       - If already latest: print "Already on the latest version (vX.Y.Z)" and return

    2. **Perform update** — use `Update::configure()` with:
       ```rust
       .repo_owner(UPDATE_REPO_OWNER)
       .repo_name(UPDATE_REPO_NAME)
       .bin_name(BIN_NAME)
       .current_version(self_update::cargo_crate_version!())
       .no_confirm(true)  // per D-04: no interactive prompt
       ```

    3. **Post-update verification** (per D-03, UPDATE-05):
       ```rust
       let exe = std::env::current_exe()?;
       let meta = std::fs::metadata(&exe)?;
       if meta.len() == 0 {
           eprintln!("error: Update produced a 0-byte binary.");
           eprintln!("This is a known issue on Windows. Please reinstall manually from:");
           eprintln!("  https://github.com/{UPDATE_REPO_OWNER}/{UPDATE_REPO_NAME}/releases/latest");
           std::process::exit(1);
       }
       ```

    4. **SHA-256 checksum verification** (UPDATE-03):
       - After update succeeds, compute SHA-256 of the new binary using `sha2` crate (already in workspace deps)
       - Fetch `checksums-sha256.txt` from the release assets via HTTP
       - Compare computed hash against the expected hash for the current platform's binary
       - If mismatch: print error with recovery hint and exit with code 1
       - Implementation note: use `reqwest::blocking` (already pulled in by self_update) to download the checksum file

    5. **Error handling** (per D-05):
       All errors must use colored, actionable messages to stderr:
       - Network timeout → "Failed to check for updates: connection timed out\nCheck your internet connection and try again."
       - HTTP 403 → "GitHub API rate limit exceeded.\nSet GITHUB_TOKEN env var for higher rate limits."
       - HTTP 404 → "No release found for your platform ({target}).\nCheck https://github.com/{owner}/{repo}/releases/latest"
       - Checksum mismatch → "Downloaded binary failed integrity check.\nTry again or download manually from {url}"
       - Already latest → "Already on the latest version (v{version})" (success, not error)

    6. **Output rules** (per D-06, ARCHITECTURE.md Pattern 3):

       - The update command itself is not run during MCP, so stdout contamination is lower risk, but `eprintln!` is still the convention for all update messages

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| binary→GitHub API | HTTP request to fetch latest release metadata and download assets |
| downloaded binary→filesystem | New binary written to disk, replacing running executable |
| user input→CLI | --check-only and --dry-run flags control update behavior |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | Downloaded binary | mitigate | SHA-256 checksum verification against checksums-sha256.txt from release assets |
| T-02-02 | Tampering | GitHub API response | accept | TLS transport (rustls) provides channel integrity; no additional signing for v2 |
| T-02-03 | Denial of Service | 0-byte binary on Windows | mitigate | Post-update binary size check (`metadata().len() == 0` → exit with error and reinstall instructions) |
| T-02-04 | Information Disclosure | Database access during update | mitigate | handle_self_update does NOT call open_store(), no SQLite connection opened |
| T-02-05 | Spoofing | GitHub API MITM | mitigate | rustls TLS certificate validation on all HTTPS requests |
</threat_model>
